# 系统升级



## 架构升级

用于解决数据量庞大的问题，解决方案如下：

- 边缘计算：在设备和云端之间，增加一个“边缘网关”，在本地进行数据预处理。主要作用：

  - 过滤数据
  - 聚合
  - 协议转换

  常用工具：EdgeX Foundry，K3s， AWS IoT Greengrass。

- 消息中间件集群（MQTT Broker Cluster）

  - 问题：用单个EMQX容器，连接数和处理能力有限
  - 解决方案：部署EMQX集群。多个EMQX节点协同工作，可以支撑百万甚至千万级的设备同时在线，并具备高可用性，一个节点挂了，其他节点还能继续服务。

- 流数据处理

  - 问题：之前java程序只是简单地把数据存起来。但是实际很多情况下，需要实时分析。比如：“过去5分钟内，A区域所有设备的平均温度是否持续上升？”
  - 解决方案：在MQTT和influxDB之间，加入一个流处理引擎。它可以对数据流进行复杂的实时计算、关联和模式管理。
  - 常用工具：Apache Flink，Apache Kafka streams， Apache NiFi。

设备 -》 边缘网关 -》 MQTT集群 -》流处理引擎 -》（InfluxDB /告警系统 / 机器学习模型）

## 安全性

需要给系统加一些安全性措施

- 设备认证：现在任何设备只要知道MQTT地址就能连接。实际场景中，每个设备都应该有唯一的身份凭证，比如X.509证书，实现双向TLS认证，确保只有合法设备才能接入。
- 通信加密：我们用的是 tcp://，数据是明文传输的。必须使用ssl://或wss://对整个通信链路进行加密，防止数据在传输过程中被窃听或篡改。
- 权限控制：设备A不应该能订阅设备B的主题。需要通过权限系统，惊喜地控制每个设备能发不（publish）和订阅（subscribe）哪些主题。

## 数据质量与治理：从“垃圾进”到“智慧出”

“垃圾进，垃圾出”是数据领域的铁律。

- 问题：传感器可能因为故障发送异常值，或者因为网络波动导致数据重复、丢失。
- 解决方案：
  - 数据清洗：在数据入库前自动过滤掉明显不合理的异常值。
  - 去重：处理因网络重试导致的重复数据。
  - 补全：对于网络中断期间丢失的数据，根据前后数据进行插值补全。
  - 统一标准：确保所有设备上报的数据格式都是统一的。

## 智能应用：从”看得见“到”看得懂“

可视化只是第一步，让数据产生真正的业务价值才是目的。

- 预测性维护：通过分析设备运行数据，利用机器学习预测设备合适可能发生故障，提前进行维护。
- 能耗优化：分析工厂或楼宇的能耗数据，自动找出能耗异常的设备或时段，并给出优化建议，甚至自动控制设备开关以节省能源。
- 数字孪生：在虚拟世界里，为物理设备创建一个完全一样的”数字双胞胎“。可以实时模拟设备的运行状态，并在虚拟环境中进行测试和调试，再反向控制物理设备。
